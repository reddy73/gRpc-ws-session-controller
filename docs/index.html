gRPC / WebSocket Session Survivability

A session-layer controller for Kubernetes that keeps long-lived gRPC streams and WebSocket connections alive across pod terminations, rolling updates, and endpoint changes.

1. Background

HTTP/1.1 vs gRPC Streams vs WebSockets

PropertyHTTP/1.1gRPC UnarygRPC StreamWebSocket
Connection lifetimePer-requestPer-requestLong-livedLong-lived
State on serverNoneNoneYes (stream context)Yes (session)
Safe to retry blindlyYesUsuallyNoNo
Affected by pod restartNoNoYesYes

When Kubernetes terminates a pod it sends SIGTERM, waits terminationGracePeriodSeconds, then sends SIGKILL. Any open TCP connections are closed. For HTTP/1.1 the client just retries. For a gRPC bidirectional stream mid-flight, the client gets an abrupt RST_STREAM or GOAWAY and has to decide: is it safe to reconnect and replay? Often it is not.

The Session Layer Gap

The OSI model has a session layer (layer 5) that manages connection state. Kubernetes operates at layer 3/4 (IP routing, TCP load balancing). There is nothing in between that understands "this TCP connection is a gRPC stream that has sent 47 messages and is waiting for message 48."

This project implements that missing session layer as a Kubernetes controller.

What "Survivability" Means Here

  • Drain — before a pod dies, tell its clients to finish up and reconnect elsewhere
  • Rebind — when a pod's IP changes (rollout), move pinned sessions to the new endpoint
  • Retry classification — know which RPCs can be safely retried vs which need dedup tokens
  • Heartbeat tracking — detect zombie sessions that lost their pod silently

3. Core Concepts

3.1 Session

A Session is the fundamental unit — one long-lived connection pinned to a specific pod.

FieldPurpose
IDUnique identifier, generated by the client or a gateway
Typegrpc or websocket
PodName / NamespaceWhich pod owns this session
EndpointPod IP:port the session is connected to
LastSeenUpdated on each heartbeat; used to detect stale sessions
RetryClassWhether this session's RPC is safe to retry

3.2 RetryClass

The most important concept for correctness. Three classes:

ClassMeaningExamples
SAFE Idempotent reads. Client can reconnect and re-subscribe freely. Watch, List, Get
CONDITIONAL Mutations with side effects. Client must send a deduplication token so the server can detect and ignore replays. Create, Update
UNSAFE Stateful streaming writes. Client MUST NOT retry without application-level coordination. Replaying messages could cause duplicates, corruption, or out-of-order processing. StreamEcho, StreamIngest

Why this mattersA naive retry-on-reconnect for an UNSAFE stream could send the same financial transaction twice, corrupt a log stream, or break a stateful protocol.

3.3 Drain vs Force-Close

terminationGracePeriodSeconds (k8s setting) │ ├── SIGTERM received │ └── DrainHandler.Handle() called │ ├── Notify all sessions (GOAWAY / WS close frame) │ ├── Poll every 500ms for sessions to self-close │ └── If deadline passes → force unregister remaining sessions │ └── SIGKILL (after grace period) └── OS closes all TCP connections

3.4 Endpoint Rebinding

During a rolling update, Kubernetes replaces pods one at a time. The Endpoints object is updated as each old pod is removed and each new pod becomes ready. The EndpointWatcher diffs old vs new subsets. When a pod IP disappears, it finds all sessions pinned to that IP and calls Rebind() to move them to a surviving IP. This is "sticky stream" behavior — the session follows the logical connection, not the physical pod IP.

3.5 Heartbeat and Stale Detection

Sessions call Heartbeat(id) periodically (e.g., every 30s). The tracker records LastSeen. Stale(ttl) returns sessions that haven't heartbeated within the TTL — these are likely orphaned (pod died without a clean drain event) and should be cleaned up.

Uses sync.RWMutex: write lock for Register/Unregister/Heartbeat, read lock for ByPod/Stale — multiple readers proceed in parallel. A plain map with RWMutex is preferred over sync.Map here because range iteration (ByPod, Stale) is awkward with sync.Map.

4.3 internal/drain/handler.go — Graceful Drain

func (h *Handler) Handle(ctx context.Context, event api.DrainEvent) {
    sessions := h.tracker.ByPod(event.PodName, event.Namespace)
    deadline := time.Now().Add(event.Deadline)

    // 1. Notify all sessions immediately
    for _, s := range sessions {
        h.notifier.Notify(s, time.Until(deadline))
    }

    // 2. Poll until drained or deadline
    ticker := time.NewTicker(500 * time.Millisecond)
    for {
        select {
        case <-ctx.Done(): return
        case <-ticker.C:
            remaining := h.tracker.ByPod(...)
            if len(remaining) == 0 { return }
            if time.Now().After(deadline) {
                for _, s := range remaining { h.tracker.Unregister(s.ID) }
                return
            }
        }
    }
}go

The DrainNotifier interface is the integration point. For gRPC: call conn.Close() or send a GOAWAY frame. For WebSocket: send a 1001 Going Away close frame. The noopNotifier in main.go just logs — replace it with your real transport logic.

4.4 internal/retry/classifier.go — Retry Safety

func (c *Classifier) Classify(method string) api.RetryClass {
    for _, r := range c.rules {
        if matchPattern(r.MethodPattern, method) {
            return r.Class
        }
    }
    return api.RetryClassUnsafe  // conservative default
}

// Default rules
{MethodPattern: "*/Watch",   Class: api.RetryClassSafe},
{MethodPattern: "*/List",    Class: api.RetryClassSafe},
{MethodPattern: "*/Create",  Class: api.RetryClassConditional},
{MethodPattern: "*/Stream*", Class: api.RetryClassUnsafe},go

Pattern matching: */Watch matches any service's Watch method. */Stream* matches any method starting with Stream. The default is UNSAFE — fail safe, not fail open.

5.2 SessionRegistry.java — Thread-Safe Registry

private final Map<String, Session> sessions = new ConcurrentHashMap<>();java

ConcurrentHashMap uses segment-level locking — concurrent reads never block each other, concurrent writes to different keys don't block each other. Compare to Go's explicit sync.RWMutex: Go gives finer control (hold the lock across multiple operations atomically); Java is simpler to write correctly.

Weakly consistent iterationConcurrentHashMap.values() may or may not reflect concurrent modifications during iteration. Acceptable here — a session registered during a drain scan is caught on the next poll cycle.

5.3 DrainHandler.java — ScheduledExecutorService

scheduler.scheduleAtFixedRate(() -> {
    List<Session> remaining = registry.byPod(podName, namespace);
    if (remaining.isEmpty()) {
        scheduler.shutdown();   // clean drain
        return;
    }
    if (Instant.now().isAfter(deadline)) {
        remaining.forEach(s -> registry.unregister(s.getId()));
        scheduler.shutdown();   // force close
    }
}, 500, 500, TimeUnit.MILLISECONDS);java

DrainNotifier is a functional interface — pass a lambda. Go uses a ticker inside a select loop tied to ctx.Done(); Java uses ScheduledExecutorService which is more idiomatic but lacks built-in context cancellation.

5.4 RetryClassifier.java — Java Records

public record Rule(String methodPattern, Session.RetryClass retryClass) {}

public static List<Rule> defaultRules() {
    return List.of(
        new Rule("*/Watch",   Session.RetryClass.SAFE),
        new Rule("*/Stream*", Session.RetryClass.UNSAFE),
        new Rule("*/Create",  Session.RetryClass.CONDITIONAL)
    );
}java

record (Java 16+) is a concise immutable data carrier — auto-generates constructor, equals, hashCode, toString, and accessors. Equivalent to a Go struct with no methods. List.of() returns an immutable list — correct for a static rule set.

5.6 SessionController.java — Entry Point

SessionController is the main class. It wires all components together and starts the controller loop.

public static void main(String[] args) {
    String namespace = System.getenv().getOrDefault("NAMESPACE", "default");

    // 1. Build components
    SessionRegistry  registry   = new SessionRegistry();
    RetryClassifier  classifier = new RetryClassifier(RetryClassifier.defaultRules());
    DrainHandler     drain      = new DrainHandler(registry, (s, ms) ->
        log.info("[notify] session=" + s.getId() + " remaining=" + ms + "ms"));

    EndpointWatcher.SessionRebinder rebinder = (s, oldEp, newEp) ->
        log.info("[rebind] session=" + s.getId() + " " + oldEp + " -> " + newEp);

    // 2. Demonstrate classifier at startup
    for (String method : new String[]{
            "/myapp.v1.MyService/Watch",
            "/myapp.v1.MyService/StreamEvents",
            "/myapp.v1.MyService/Create"}) {
        log.info("method=" + method + " retryClass=" + classifier.classify(method));
    }

    // 3. Register a demo session
    Session demo = new Session();
    demo.setId("demo-session-1");
    demo.setType(Session.Type.GRPC);
    demo.setPodName("myapp-pod-abc");
    demo.setNamespace(namespace);
    demo.setEndpoint("10.0.0.1:50051");
    registry.register(demo);

    // 4. Connect to Kubernetes and start endpoint watcher
    try (KubernetesClient client = new KubernetesClientBuilder().build()) {
        // auto-detects: in-cluster token OR ~/.kube/config
        new EndpointWatcher(registry, rebinder).watch(client, namespace);

        Thread.sleep(5000);  // demo: simulate drain after 5s
        drain.handle("myapp-pod-abc", namespace, 30_000);

        Thread.currentThread().join();  // block until interrupted
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
}java

KubernetesClientBuilder().build() auto-detects configuration in order:

  1. In-cluster: reads /var/run/secrets/kubernetes.io/serviceaccount/token and the cluster CA cert
  2. Out-of-cluster: reads ~/.kube/config (same as kubectl)

The try-with-resources ensures the HTTP client and watch streams are closed cleanly on exit. Thread.currentThread().join() blocks the main thread indefinitely — the endpoint watcher runs on fabric8's internal thread pool.

5.7 Threading Model

Understanding which thread runs what is critical for avoiding races in the Java implementation.

Thread: main └── SessionController.main() ├── registry.register() ← ConcurrentHashMap write, thread-safe ├── classifier.classify() ← stateless, no sync needed ├── drain.handle() ← submits to scheduler thread pool └── client.watch() ← returns immediately, watch on fabric8 thread Thread: fabric8-watcher-0 (fabric8 internal) └── EndpointWatcher.eventReceived() └── reconcile() ├── registry.byPod() ← ConcurrentHashMap read, thread-safe └── rebinder.rebind() ← your callback, runs on this thread Thread: pool-1-thread-1 (ScheduledExecutorService) └── DrainHandler lambda (every 500ms) ├── registry.byPod() ← ConcurrentHashMap read, thread-safe └── registry.unregister() ← ConcurrentHashMap write, thread-safe

No explicit synchronization needed because ConcurrentHashMap handles all cross-thread access to the session store. The only non-thread-safe field is EndpointWatcher.lastKnownAddresses (a plain HashMap) — safe because fabric8 delivers all events on a single watcher thread.

8. Key Design Decisions

Why interfaces for DrainNotifier and SessionRebinder?

The controller doesn't know how your sessions are implemented. You might use grpc.ClientConn.Close() for gRPC channels, websocket.Conn.WriteMessage(websocket.CloseMessage, ...) for WebSockets, or a message queue notification for async clients. By depending on an interface, the controller is transport-agnostic.

Why poll instead of event-driven drain?

Sessions might crash without calling Unregister. The deadline must be enforced regardless of session behavior. Polling is simple and predictable; event-driven would require sessions to signal back. The 500ms interval is a tradeoff — fast enough to react quickly, slow enough not to hammer the tracker's mutex.

Why UNSAFE as the default RetryClass?

An unclassified stream is assumed to be stateful and not safe to retry. You must explicitly opt in to SAFE or CONDITIONAL by adding a rule. The alternative (default to SAFE) would silently allow retries on streams that might cause data corruption. Fail safe, not fail open.

Why kind over Docker Desktop's built-in Kubernetes?

Docker Desktop ships with a single-node cluster. kind gives you multi-node clusters (test real cross-node scheduling), multiple independent clusters (test multi-cluster scenarios), reproducible cluster configs in YAML, and easy teardown and recreation.

Troubleshooting

Pod stuck in ImagePullBackOff

imagePullPolicy: Never requires the image to be loaded via kind load.

kind load docker-image echoserver:test --name session-cluster-a
kind load docker-image session-controller:test --name session-cluster-abash
Controller crashes with "k8s config: ..."

The controller uses InClusterConfig — it must run inside the cluster. For local testing outside a cluster, switch to kubeconfig:

cfg, err := clientcmd.BuildConfigFromFlags("", os.Getenv("KUBECONFIG"))go
rollout status times out 
kubectl --context kind-session-cluster-a -n session-test describe podsbash

Check pod events for the real error — usually image pull, resource limits, or readiness probe failure.

grpc_health_probe not found 
go install github.com/grpc-ecosystem/grpc-health-probe@latest
export PATH=$PATH:$(go env GOPATH)/binbash
kind cluster creation fails

Ensure Docker Desktop is running with enough resources (recommended: 4 CPU, 8GB RAM).

docker info | grep -E 'CPUs|Total Memory'bash

Expected on Kubernetes 1.33+. The watcher works but for production migrate to EndpointSlice:
